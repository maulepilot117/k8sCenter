Editing Experience — Two Options

When you click a resource row or "New Certificate", how should editing work?

A

Schema-Driven Form + YAML Toggle

Parse the CRD's OpenAPI validation schema (from the CRD spec) and auto-generate a structured form. String fields become text inputs, enums become dropdowns, arrays become repeatable sections, nested objects become collapsible fieldsets.

A toggle button switches between Form view and raw YAML (Monaco editor). Changes in either view sync to the other. YAML preview always available before apply.

Pros

  • Structured editing without knowing YAML structure
  • Validation happens as you type (from CRD schema)
  • Discoverable — users can see all available fields
  • YAML escape hatch always available

Cons

  • Complex to implement (recursive schema → form generation)
  • Some CRDs have poor/missing schemas
  • Deeply nested schemas produce unwieldy forms
B

Smart YAML Editor with Schema Assist

Recommended. Primary editing in Monaco YAML editor with schema-powered intelligence: autocomplete from the CRD's OpenAPI schema, inline validation, hover documentation for fields, and error squiggles for invalid values.

A collapsible schema sidebar shows the CRD's field structure as a tree — click any field to insert it at cursor. The sidebar also shows which fields are required vs optional, with descriptions pulled from the schema.

Top bar has: Validate (dry-run), Diff (show changes vs current), Apply (server-side apply).

Pros

  • Consistent with existing YAML tools page pattern
  • Works great even with poor/missing schemas (graceful degradation)
  • Schema sidebar gives discoverability without form complexity
  • Power users stay in YAML, beginners use sidebar for guidance
  • Simpler to implement — extends existing Monaco integration

Cons

  • Less approachable for users unfamiliar with YAML
  • No visual form for simple field edits
