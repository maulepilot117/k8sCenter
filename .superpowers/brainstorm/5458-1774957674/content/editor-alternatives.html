YAML Editing — Alternatives to Monaco

Monaco had issues in this project. Here are three lighter approaches.

A

CodeMirror 6 + YAML Language Support

CodeMirror 6 is a ground-up rewrite — lightweight, modular, and works well with Preact/islands. It has first-class YAML syntax highlighting via @codemirror/lang-yaml, and the extension system supports custom linting, autocomplete, and bracket matching.

~50KB gzipped (vs Monaco's ~2MB). No Web Worker required. Works on mobile. The API is functional — no class-based editor instances to manage.

Pros

  • 10-20x smaller than Monaco — fast load, no bundle bloat
  • Modular — only import what you need
  • Proven in production (used by Chrome DevTools, Replit, Observable)
  • Easy to add custom linting from CRD schema validation
  • Works in SSR/islands architecture without hacks

Cons

  • No built-in YAML schema validation — need custom linter extension
  • Different API from Monaco (learning curve if familiar with Monaco)
B

Schema-Driven Form (No Code Editor)

Recommended. Skip the code editor entirely for CRD editing. Instead, auto-generate a structured form from the CRD's OpenAPI validation schema. Text inputs for strings, dropdowns for enums, toggles for booleans, repeatable rows for arrays, collapsible sections for nested objects.

A "View YAML" tab shows the generated output as read-only syntax-highlighted YAML (using a simple <pre> with Prism.js or even just CSS — no editor needed). Users who need raw YAML can use the existing YAML Tools page.

This is the "modern approach" you were asking for — most k8s tools show raw YAML, but a form-first approach is what makes CRD management actually accessible.

Pros

  • Most approachable — no YAML knowledge required
  • Inline validation per field (type, required, enum, pattern)
  • Discoverable — all fields visible with descriptions from schema
  • Consistent with existing wizard pattern (WizardStepper → steps → review)
  • Zero editor dependencies — just native HTML form elements
  • YAML preview is read-only, so no editor complexity

Cons

  • Complex schemas produce deep forms (mitigated by collapsible sections)
  • CRDs without schemas degrade to a key-value editor or raw textarea
  • Can't paste raw YAML directly (use YAML Tools page for that)
C

Hybrid: Form Primary + Syntax-Highlighted Textarea Fallback

Schema-driven form (like B) as the default editing experience. For CRDs without schemas, or when the user clicks "Edit as YAML", fall back to a syntax-highlighted <textarea> using a lightweight library like highlight.js overlay or CSS-only highlighting.

No heavy editor. Just a styled textarea with line numbers and basic YAML color coding. Validate/Diff/Apply buttons work the same way regardless of mode.

Pros

  • Form-first for structured editing, raw YAML for power users
  • Graceful degradation for schema-less CRDs
  • Lightweight — no editor bundle at all
  • Users can paste YAML directly in textarea mode

Cons

  • Textarea YAML editing is basic (no autocomplete, no bracket matching)
  • Two editing modes to maintain
