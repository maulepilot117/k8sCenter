CRD Management: Three Approaches

How should custom resources fit into k8sCenter's navigation and UX?

A

Extensions Domain — Dedicated 9th Section

Add a new "Extensions" icon rail section alongside Workloads, Networking, etc. Auto-discovers all CRDs at startup and groups them by API group (e.g., cilium.io, cert-manager.io, argoproj.io). Each API group becomes a collapsible section with its resource types as tabs.

Users can pin frequently-used CRDs to promote them into existing domain sections (e.g., pin Certificates into Security). Pinned resources appear as native tabs alongside built-in resources.

Pros

  • Clean separation — extensions don't clutter built-in sections
  • Natural home for discovery ("what CRDs exist?")
  • Pinning gives power users shortcuts without forcing organization
  • API group clustering gives immediate context

Cons

  • CRDs live "away" from related built-in resources by default
  • Extra click to reach extensions vs. built-in resources
  • Pinning adds UI complexity (pin management, sync state)
B

Smart Routing — Auto-classify CRDs into Existing Domains

No new section. Instead, k8sCenter intelligently classifies CRDs into existing domains based on API group heuristics. cilium.io → Networking, cert-manager.io → Security, snapshot.storage.k8s.io → Storage, etc. Unclassifiable CRDs go into an "Other" tab in a lightweight overflow section.

An "Extensions" overview page (accessible from the icon rail) shows ALL CRDs in one place for discovery, but the day-to-day tabs live within their natural domain.

Pros

  • CRDs feel native — Certificates next to Secrets, Cilium policies next to NetworkPolicies
  • No context switching between "built-in" and "custom"
  • Discovery page still exists for exploration
  • Matches mental model ("this is a networking thing")

Cons

  • Classification heuristics can be wrong — user override needed
  • Domain sections could get crowded with many CRDs
  • New operators might not map cleanly to existing domains
C

Hybrid Hub — Extensions Section + Smart Suggestions

Recommended. Combines A and B. A dedicated "Extensions" section in the icon rail serves as the CRD hub — auto-discovers everything, groups by API group, shows CRD metadata (scope, versions, schema). From here, users can assign CRDs to existing domains with one click.

k8sCenter suggests domain assignments based on API group (e.g., "Assign certificates.cert-manager.io to Security?") but never forces it. Assigned CRDs appear as tabs in their domain. Unassigned CRDs remain accessible in Extensions.

The Extensions hub also shows a CRD health dashboard — which CRDs are installed, their versions, how many instances exist, and whether the backing operator is healthy.

Pros

  • Best of both: dedicated discovery + contextual integration
  • User controls organization — suggestions, not assumptions
  • CRD health dashboard adds operational value beyond just browsing
  • Scales from 5 to 50 CRDs without getting overwhelming

Cons

  • Most complex to implement
  • Assignment state needs persistence (database or ConfigMap)
  • Suggestion engine needs heuristic tuning
